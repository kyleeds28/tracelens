# 산출문서 ↔ 코드 검증 시스템 가이드

> 산출문서(엑셀)에 적힌 모듈 목록과 실제 소스 코드가 **얼마나 일치하는지**
> 자동으로 대조해주는 도구입니다. 비개발자도 한 화면에서
> "어디가 일치하고, 어디가 잘못됐는지" 파악할 수 있습니다.

## 1. 한눈에 보기 — 이 시스템이 하는 일

**비유**: 책의 **목차(산출문서)** 와 **실제 본문(소스 코드)** 을 한 줄씩 대조해서
"목차에 적힌 항목이 본문에 실제로 있는지, 내용이 맞는지" 확인하는 검수원.

### 입력

```
📄 산출문서 (xlsx)           : KGD-DS-14_프로그램목록_xxx.xlsx
📦 소스 코드 (zip)           : portal_backend-master.zip
```

### 출력 (화면에 보이는 것)

```
✅ 일치       : 산출문서에 적힌 모듈이 코드에도 같은 위치에 있음
⚠️ 부분 일치  : 모듈 이름은 같은데 패키지(폴더)가 다름
❌ 불일치     : 코드에서 못 찾음 (오타 또는 삭제됨)
```

추가로:
- 메서드별 구현 상태 (정상 / 의심 / 알 수 없음)
- 산출문서 수정 권고 (어디를 어떻게 바꾸면 되는지 친절히)
- (선택) AI 가 본문 직접 읽고 의도 검증

## 2. 왜 필요한가

전형적 시나리오: 개발 완료 후 **산출문서가 코드 변경을 못 따라가는 경우** 가 흔합니다.

| 흔한 상황 | 이 도구가 발견 |
|---|---|
| 개발 중 클래스를 `dashboard.repository` → `datastatus.repository` 로 이동 | 산출문서는 옛 패키지 그대로 → "패키지 정정 필요" 자동 표시 |
| 산출문서엔 `UserService` 라고 있는데 실제 코드엔 없음 | "모듈 누락" 으로 빨간색 표시 |
| `return null;` 만 있는 미구현 메서드 | "임시 반환값" 으로 의심 마크 |
| Spring Security 처럼 자동 호출되는 메서드를 "안 쓰임" 으로 잘못 보고됨 | 분석기가 그런 패턴을 인식해서 정상으로 분류 |

수작업으로 1000+ 메서드를 검토하면 며칠~몇 주, 이 도구로 약 30초~3분.

## 3. 사용 흐름 (5단계)

```
1. 산출문서(xlsx) 업로드
       ↓ 시트·헤더 자동 감지, 컬럼 매핑
2. 소스 코드(zip) 업로드  
       ↓ 압축 해제, 파일 인덱싱
3. 자동 매칭 수행
       ↓ 일치/부분/불일치 산정
4. 행 클릭으로 상세 분석 (선택)
       ↓ tree-sitter 가 클래스/메서드 분해
5. (옵션) AI 검증
       ↓ 회색지대 메서드에 LLM 호출
```

각 단계는 다음 절에서 자세히 다룹니다.

### 단계 1: 산출문서(xlsx) 업로드

엑셀 양식이 회사/프로젝트마다 다를 수 있습니다 (KGD-DS, MEBP, EOS …).
**자동으로 어느 시트·어느 행이 헤더인지 판별** 합니다.

자동 인식 룰:
- 시트 이름이 `프로그램목록` / `프로그램 목록` / `Program List` 등 변형 → 자동 매칭
- 헤더가 2행/3행/4행 어디 있어도 → 컬럼 매칭 점수 가장 높은 행 자동 선택
- 한 컬럼에 여러 이름 인정: 예) `프로그램ID`, `프로그램ID(오퍼레이션)`, `프로그램 ID` 모두 같은 의미로

설정 파일 [`mapping.yaml`](../app/backend/mapping.yaml) 에 변형들이 등록되어 있고,
새 양식 추가 시엔 한 줄만 추가하면 됩니다.

### 단계 2: 소스 코드(zip) 업로드

zip 파일을 압축 해제하고 파일을 인덱싱합니다.

- 지원 확장자: `.java`, `.kt`, `.py`, `.js`, `.jsx`, `.ts`, `.tsx`, `.vue`
- 자바 파일은 `package` 선언을 자동 파싱해서 **FQCN (Fully Qualified Class Name)** 생성
  - 예) `com.kyobo.portal.user.UserService`

### 단계 3: 자동 매칭

산출문서 한 행마다 코드의 해당 파일을 찾습니다.
규칙은 [§5. 매칭은 어떻게 산정되나](#5-매칭은-어떻게-산정되나) 참고.

### 단계 4: 상세 분석

행의 `분석 ▾` 버튼 클릭 시:
- tree-sitter 가 그 파일을 파싱해서 클래스/메서드 목록·메트릭 표시
- (Java) JavaParser+SymbolSolver 가 호출 그래프 정보 추가

### 단계 5: AI 검증 (선택)

정적 분석이 자신 없는 메서드(회색지대) 에만 `🤖 AI 분석` 버튼이 나옴.
클릭 시 로컬 LLM(`qwen2.5-coder:7b`) 이 본문을 읽고 의도를 판단.

## 4. 핵심 도구 3가지 — 비유로 이해하기

이 시스템은 **3개의 분석 도구가 단계적으로 협력**합니다. 각자 잘하는 게 다릅니다.

### ① tree-sitter — "구문 분해기" (편집기 수준)

**비유**: 책을 받아서 **목차·문장 단위로 분해**해주는 도구.
의미는 안 보고 구조만 빠르게 봅니다.

**잘하는 것**:
- 매우 빠름 (수천 파일을 초 단위로 파싱)
- **여러 언어 지원** (Java, Python, JS, TS, Vue 등)
- 컴파일 안 되어도 동작 (구문 에러 있어도 부분 파싱)
- 클래스/메서드 추출, 어노테이션 식별, 줄 수·복잡도 계산

**못하는 것 (한계)**:
- "이 변수의 진짜 타입이 뭐냐?" 같은 의미 추론 불가
- `userService.foo()` 라는 코드를 보면 "foo 라는 메서드를 부른다" 까지만 알고
  → **어느 클래스의 어느 foo 인지는 모름**

**비유로 정리**:
> 도서관 사서가 책을 받자마자 "이 책엔 챕터가 N개, 각 챕터마다 단락이 M개"
> 라고 빠르게 알려줍니다. 하지만 챕터 내용까지는 안 읽습니다.

### ② JavaParser — "자바 전용 정밀 파서"

**비유**: **자바 문법 전문 사전**. 자바 코드를 자바 컴파일러가 보는 것처럼 깊게 이해.

**잘하는 것**:
- 자바의 모든 문법 요소를 정확히 파악 (제네릭, 람다, 어노테이션, 인터페이스 …)
- 클래스 계층, 메서드 시그니처, 필드 선언을 정확히 매핑

**못하는 것**:
- 자바만 다룸 (Python/JS 분석 불가 — tree-sitter 가 처리)
- 자체로는 의미 추론 안 함 (그건 SymbolSolver 가 추가됨)

### ③ SymbolSolver — "타입 추적기"

**비유**: **단어들이 정확히 어느 사전 항목을 가리키는지** 따라가주는 색인 시스템.

**잘하는 것**:
- `userRepository.findByEmail(x)` 같은 호출의 **진짜 풀네임** 결정:
  `com.example.UserRepository.findByEmail(String)`
- 누가 누구를 부르는지 (호출 그래프) 구축
- 인터페이스 ↔ 구현체 연결 (`UserRepositoryCustom` → `UserRepositoryImpl`)

**못하는 것 (한계)**:
- **외부 라이브러리는 못 봄** (Spring, Lombok, JDK 등 jar 미등록)
  - 그래서 `log.info(...)` 같은 호출은 "외부" 로 표시되고 풀 시그니처 추적 안 됨
- **Lombok 자동생성 메서드 못 봄** (소스에 없고 컴파일 후에만 존재)
- 우리 프로젝트 안 호출이라도 중간에 외부 인터페이스 거치면 막힘

**비유로 정리**:
> "철수가 영희를 만났다" 라는 문장에서 **철수와 영희가 정확히 누구인지** (성씨, 주소)
> 까지 추적해줍니다. 다만 **외국인 등장인물** (외부 lib) 은 우리 주민등록부에 없어서 못 찾음.

### 3단계 협력 흐름

```
[1단계] tree-sitter  → 모든 파일 빠르게 파싱
                       "이 파일에 클래스 N개, 메서드 M개 있음"
                       "각 메서드의 줄 수, 복잡도, 호출 이름"

[2단계] JavaParser   → 자바 파일만 다시 깊이 파싱
                       "클래스 간 상속·구현 관계"
                       "메서드 시그니처 정확히"

[3단계] SymbolSolver → 호출 그래프 빌드
                       "이 호출이 어느 메서드를 가리키는지"
                       "누가 누구를 부르는지"
```

## 5. 매칭은 어떻게 산정되나

산출문서 한 행 (`UserController` 모듈) 을 받아서:

```
1. 산출문서에 적힌 정보 추출
   ┌─ package    : com.kyobo.portal.user.controller
   ├─ module_name: UserController
   └─ kind       : api (또는 front)

2. 기대 풀네임 만들기
   expected = "com.kyobo.portal.user.controller.UserController"

3. 코드에서 찾기
   3-1. FQCN 정확 매치 → 일치 (✅ O)
   3-2. 이름만 매치, 패키지 다름 → 부분 일치 (⚠ PARTIAL)
   3-3. 어디서도 못 찾음 → 불일치 (❌ X)
```

### 매칭 결과 3가지

| 상태 | 의미 | UI 표시 |
|---|---|---|
| **일치 (O)** | 패키지 + 이름 모두 일치 — 완벽 매칭 | 초록 |
| **부분 일치 (PARTIAL)** | 이름은 같지만 패키지가 다름. 코드 이동/리네이밍 가능성 | 노랑 |
| **불일치 (X)** | 같은 이름 클래스가 어디에도 없음. 삭제됐거나 이름 변경됨 | 빨강 |

### 예시

산출문서:
```
package    : com.kyobo.portal.content.dashboard.repository
module_name: MetaDataTxnRepository
```

실제 코드:
```
파일 위치: portal_backend/src/main/java/.../datastatus/repository/MetaDataTxnRepository.java
실제 FQCN: com.kyobo.portal.content.datastatus.repository.MetaDataTxnRepository
```

→ **이름 같음, 패키지만 다름** → `부분 일치 (PARTIAL)` → 시스템이 "산출문서 패키지를
`datastatus` 로 정정 권장" 안내.

### 매칭 전략 (자동)

| kind | 사용 전략 | 확장자 |
|---|---|---|
| `api` (백엔드) | FQCN 정확 매칭, 실패 시 simple_name 매칭 | `.java`, `.kt` |
| `front` (프론트) | 파일명 매칭 | `.vue`, `.tsx`, `.jsx`, `.ts`, `.js` |
| 미지정 | 어느 확장자든 이름 매칭 | 전체 |

## 6. 메서드 단위 상태 판정

매칭된 파일을 열면 각 메서드마다 상태가 표시됩니다.

### 3가지 상태 그룹

| 그룹 | 색 | 의미 | 예시 verdict |
|---|---|---|---|
| **정상 (ok)** | 초록 | 구현이 적절한 형태 | 구현됨 / 컨트롤러 위임 / 서비스 위임 / 필드 접근자 |
| **의심 (suspect)** | 빨강 | stub 또는 비정상 패턴 | 임시 반환값 / 빈 본문 / 미구현 예외 |
| **알 수 없음 (unknown)** | 회색 | 정적 분석이 분류 못 함 | 한 줄 본문 (분류 보류) |

### verdict 종류 (요약)

| verdict | 한국어 라벨 | 의미 |
|---|---|---|
| `HANDLER_DELEGATION` | 컨트롤러 위임 | Controller 가 Service 호출만 하는 정상 패턴 |
| `HANDLER_PROCESSING` | 컨트롤러 처리 | Controller 가 validation+위임 정도 |
| `SERVICE_DELEGATION` | 서비스 위임 | Service 가 다른 Service/Repository 호출 |
| `DELEGATION` | 메서드 위임 | 일반 위임 |
| `ACCESSOR` | 필드 접근자 | getter/setter 같은 단순 필드 반환 |
| `LOGIC_LIGHT` | 구현됨 (단순) | 5~14줄 정도의 본문 로직 |
| `LOGIC_NORMAL` | 구현됨 (보통) | 15~39줄, 분기 있음 |
| `LOGIC_COMPLEX` | 구현됨 (복잡) | 40줄+ 또는 cc 8+ |
| `STUB_NOT_IMPL` | 미구현 예외 | `throw new UnsupportedOperationException()` |
| `STUB_PLACEHOLDER` | 임시 반환값 | `return null;` / `return false;` 등 |
| `STUB_EMPTY` | 빈 본문 | `{ }` 만 있음 |
| `STUB_DEBUG_ONLY` | 디버그 출력만 | `log.info(...)` 만 있음 |
| `GENERATED` | 프레임워크 자동생성 | MapStruct/MyBatis Mapper 등 |
| `DECLARATION` | 인터페이스 선언 | Spring Data JPA Repository 인터페이스 |
| `ABSTRACT` | 추상 메서드 | 본문 없음 (인터페이스/abstract) |
| `STRAIGHT_LINE` | 한 줄 본문 (정적 분석 분류 보류) | 분류 못 함 — 검토 권장 |

### 메서드 표 컬럼 의미

| 컬럼 | 의미 |
|---|---|
| **메서드** | 자바 메서드 이름 (+ 🤖? = AI 검증 권장 마커) |
| **상태** | 정상/의심/알수없음 + 라벨 |
| **layer** | 클래스 종류 (CONTROLLER/SERVICE/REPOSITORY/UTIL/...) |
| **어노테이션** | `@PostMapping`, `@Override` 등 |
| **본문** | `sloc` (실 코드 줄) / `cc` (복잡도) / `stmts` (문장 수) |
| **호출** | `out` (이 메서드가 부르는 함수 수) / `in` (이 메서드를 부르는 곳 수) |
| **근거 / 위임 대상** | 진짜 어디로 가는지 (resolved 호출 또는 정적 근거) |
| **lines** | 본문 줄 수와 위치 |

### `out` 과 `in` 의 정확한 의미

| | **out (fan-out)** | **in (fan-in)** |
|---|---|---|
| 의미 | 이 메서드가 **부르는** 함수 수 | 이 메서드를 **부르는** 다른 코드 위치 수 |
| 출처 | tree-sitter (즉시 카운트) | SymbolSolver (프로젝트 전체 스캔) |
| 외부 lib 포함? | ✅ 포함 (이름만 보면 됨) | ❌ 외부 jar 소스 없어서 못 셈 |

**중요**: `in=0` 이 "이 메서드 안 쓰임" 의미는 아님.
HTTP 진입점 / Spring 이벤트 / reflection 호출은 정적 분석에 안 잡힘.

### "N/M 호출 해석" 의 의미

본문에 호출이 M개 있는데 SymbolSolver 가 N개 해석함:
- `8/8 해석` = 모두 우리 코드/JDK 호출. 추적 완료.
- `0/20 해석` = 모두 외부 lib (Spring/Lombok 등). 추적 불가.
- `3/15 해석` = 일부만 우리 코드. 나머지는 외부.

해석된 호출들은 표에 "→ TargetClass.method() [LAYER]" 로 표시됩니다.

## 7. 산출문서 정합성 자동 검사

각 행의 매칭 결과를 분석해서 **사용자 친화적 수정 권고** 를 자동 생성:

### 3가지 이슈 종류

| 종류 | 한국어 라벨 | 무엇 |
|---|---|---|
| `PACKAGE_DRIFT` | 패키지 정정 필요 | 모듈은 있는데 패키지가 다름. 코드 이동/리네이밍 가능 |
| `AMBIGUOUS` | 후보 다수 | 같은 이름의 클래스가 여러 패키지에 존재 — 어느 것 가리키는지 명시 필요 |
| `MODULE_MISSING` | 모듈을 찾을 수 없음 | 코드에 아예 없음. 삭제됐거나 이름 변경됨 |

### 표시 위치

- 모듈 행을 **분석 ▾** 으로 펼치면 패널 최상단에 **노란색 권고 카드** 자동 표시
- "산출문서엔 X 패키지인데 실제는 Y 패키지" 식 before/after 명시
- **[복사]** 버튼으로 실제 패키지명 클립보드 복사 → xlsx 셀에 바로 붙여넣기

### CSV 다운로드

전체 이슈를 한 번에 정리하려면 프로젝트 요약의 **📥 CSV 다운로드** 클릭.
컬럼: `row_idx, kind, severity, program_id, program_name, module_name,
spec_package, actual_package, fix_column, fix_from, fix_to, suggestion`.

UTF-8 BOM 포함이라 엑셀에서 한글 깨짐 없음.

## 8. AI 정밀 분석 (선택)

### 회색지대 자동 식별

정적 분석으로 확신 못 하는 메서드에만 **🤖? 마커** 표시. 4가지 회색지대:

1. **status='unknown'** — 정적 분석 분류 보류
2. **status='suspect'** — stub 의심 (AI 2차 의견)
3. **위임 판정 + 호출 대상 미해석** — SymbolSolver 가 추적 끊김
4. **status='ok' + 호출 0% 해석** — 본문엔 호출이 있는데 모두 외부 lib

### LLM 호출 흐름

```
사용자가 [🤖 AI 분석] 클릭
    ↓
프롬프트 생성 (tree-sitter + SymbolSolver 결과 + 산출문서 명세 + 본문 코드)
    ↓
로컬 Ollama (qwen2.5-coder:7b) 호출 (sequential, ~10~20초)
    ↓
결과 받음 → DB 캐시 → 화면 표시
```

### 5가지 verdict

| verdict | 한국어 라벨 | 의미 |
|---|---|---|
| `REAL_DELEGATION` | 실제 위임 | 다른 메서드에 일을 맡김 |
| `REAL_LOGIC` | 실제 로직 | 본문에서 직접 검증/계산 |
| `STUB` | 미구현 stub | `return null` / `throw NotImpl` 등 |
| `NAME_MISMATCH` | 이름·본문 불일치 | 메서드명이 약속하는 일을 본문이 안 함 |
| `UNCLEAR` | 판단 모호 | 신뢰도 낮을 때 |

### 결과 패널 동작

```
┌─ AI 결과 ─────────────────────────────┐
│ [실제 로직] 신뢰도 90%             ↺ │  ← ↺ 재분석
│ "본문에서 직접 검증 로직 수행"        │
│ 💡 추정 역할: 인증 객체 null 체크      │
│ ─────────────────────────────────────│
│ [적용] [취소]                         │
└──────────────────────────────────────┘
```

| 버튼 | 동작 | 새로고침 후 |
|---|---|---|
| **[적용]** | LLM 판정을 메서드 상태로 승격 + AI 마크 🤖90% 추가 | 적용 상태 유지 (DB 저장) |
| **[취소]** | DB 캐시 영구 삭제 + 화면 초기화 | 초기 [🤖 AI 분석] 버튼 |
| **`적용 취소`** (적용 후) | applied 만 false (결과는 유지) | 결과 패널 + [적용][취소] 다시 |
| **`↺`** | 강제 재분석 (캐시 무시) | 새 결과 |

### 신뢰도 90% 캡

7B 모델은 종종 100% 자신감을 표현 — 실제로는 과신.
**모든 LLM 응답의 confidence 를 90% 로 제한** 표시.

### 하드웨어 보호 (8GB VRAM 룰)

- 모든 LLM 호출은 백엔드에서 **mutex 로 직렬화** (한 번에 1건만 GPU 점유)
- UI 에서도 한 메서드 분석 중엔 다른 버튼들이 `⌛ 대기` 로 비활성화
- CPU offload 위험 없음

## 10. 자주 묻는 질문 (FAQ)

### Q. "왜 다 PARTIAL 인가요?"

산출문서가 패키지 정보를 안 적었거나, 모듈명만 적었을 때 발생.
또는 코드가 산출문서 작성 후 이동된 경우.

→ **각 행의 산출문서 권고 카드** 확인. 자동으로 "X 패키지로 정정" 안내.

### Q. "이 메서드 분명 쓰는데 `in=0` 으로 보입니다"

`in` 은 **정적 자바 호출 그래프만** 추적. 다음 케이스는 항상 `in=0`:
- HTTP 진입점 (`@GetMapping` Controller 메서드) — Spring 이 reflection 으로 호출
- Spring 이벤트 핸들러 (`@EventListener`)
- 스케줄러 (`@Scheduled`)
- 인터페이스 구현 메서드 (호출이 인터페이스 거치면 추적 끊김)

→ `in=0` 은 **"정적 분석에 안 잡힘"** 이지 **"안 쓰임"** 아님.

### Q. "Lombok DTO 가 메서드 0개로 표시됩니다"

`@Getter @Setter @Builder @AllArgsConstructor` 등은 컴파일 시 메서드 자동 생성.
**소스에는 안 보임**. 화면에 `🔧 Lombok` 칩 + 안내 노트 자동 표시.

DTO/Entity 의 정상 패턴 — 산출문서 검증상 문제 없음.

### Q. "0/8 호출 해석 이라는 게 뭔가요?"

본문에 8개 호출이 있는데 SymbolSolver 가 0개 풀어냄 = **본문 호출 추적 끊김**.
대부분 Spring/Slf4j/JDK 같은 외부 lib 호출이라 그렇습니다.

회색지대로 인식되어 `🤖 AI 분석` 버튼 자동 노출 → 본문 의미 LLM 으로 검증 가능.

### Q. "AI 분석은 얼마나 정확한가요?"

- **verdict (5지선다 분류)**: 80~90% 정확
- **신뢰도 숫자**: 항상 90% 로 cap (7B 모델 과신 보정)
- **도메인 깊이**: 7B 한계 — 흔한 Spring 패턴은 알지만 미세한 의도까지는 부족
- **권장**: AI 결과를 참고로, 확신 안 들면 `적용` 안 하고 사람이 코드 확인

### Q. "AI 가 stub 이라고 했는데 사실은 정상 구현이면요?"

[취소] 버튼으로 결과 삭제, 또는 [적용] 안 하고 패널 닫기.
정적 분석 결과는 그대로 유지됩니다.

### Q. "왜 'UNKNOWN' 이라는 layer 가 안 보이고 'Utility' / 'Manager' 가 보이나요?"

이전엔 분석기가 못 알아본 클래스를 모두 `UNKNOWN` 으로 표시했으나,
현재는 **클래스 이름의 suffix 그대로** 표시 (예: `StringUtility` → `Utility`).
정직한 정보 우선 정책.

### Q. "산출문서를 수정한 뒤 다시 매칭하려면?"

사이드바의 **[re-match]** 버튼. zip 재업로드 없이 매칭만 재실행.

### Q. "AI 분석 결과는 새로고침 후에도 유지되나요?"

- **적용한 결과**: DB 저장 → 유지 ✅
- **[취소] 한 결과**: DB 삭제 → 새로고침 후에도 사라짐 ✅
- **그냥 분석만 한 결과 (적용 안 함)**: DB 캐시 → 새로고침 시 결과 재출현 (적용/취소 다시 선택 가능)

## 11. 용어 사전

| 용어 | 의미 |
|---|---|
| **모듈** | 산출문서 한 행 (= 코드 한 파일에 매칭되는 단위) |
| **프로그램** | 여러 모듈을 묶은 단위 (산출문서의 `program_id`) |
| **FQCN** | Fully Qualified Class Name. 패키지 + 클래스명. 예) `com.example.UserService` |
| **fqsig** | Fully Qualified Signature. FQCN + 메서드명 + 파라미터. 예) `com.example.UserService.foo(String)` |
| **AST** | Abstract Syntax Tree. 소스코드를 트리로 분해한 구조 |
| **tree-sitter** | 다국어 지원 빠른 파서 (구조만 봄) |
| **JavaParser** | 자바 전용 정밀 파서 |
| **SymbolSolver** | JavaParser 의 타입/호출 추적 모듈 |
| **fan-out** | 이 메서드가 **부르는** 함수 수 |
| **fan-in** | 이 메서드를 **부르는** 다른 위치 수 |
| **SLOC** | Source Lines Of Code. 빈 줄·주석 제외한 코드 줄 수 |
| **CC** | Cyclomatic Complexity. 분기·단락평가의 수. 1이면 분기 없음, 높을수록 복잡 |
| **stmts** | 본문의 명령(statement) 개수 |
| **layer** | 클래스의 역할 분류 (CONTROLLER/SERVICE/REPOSITORY/MAPPER/ENTITY/DTO/UTIL/CONFIG/COMPONENT) |
| **verdict** | 분석기가 내린 메서드의 판정 코드 (HANDLER_DELEGATION, LOGIC_LIGHT 등) |
| **회색지대** | 정적 분석이 자신 못 하는 메서드 (LLM 검증 권장) |
| **LLM** | Large Language Model. 우리는 로컬 `qwen2.5-coder:7b` 사용 |
| **Lombok** | 자바에서 getter/setter/생성자를 어노테이션으로 자동 생성해주는 라이브러리 |
| **Spring** | 자바 백엔드의 표준 프레임워크 |
| **JPA** | Java Persistence API. DB ↔ 객체 매핑 표준 |
| **MyBatis** | SQL 매핑 라이브러리 |
| **QueryDSL** | 자바 코드로 타입 안전 쿼리 작성하는 라이브러리 |

  │ 4. 사용자가 모듈 행 "분석 ▾" 클릭
  ▼
[backend: ast_analyzer.py]
  │ - tree-sitter 로 해당 파일 파싱
  │ - 메서드별 verdict 산출
  │ - 캐싱 (mapping.ast_json)
  ▼
[화면: 메서드 상태/메트릭/근거 표시]
  │
  │ 5. (회색지대) 사용자가 "🤖 AI 분석" 클릭
  ▼
[backend: llm_analysis.py → Ollama → qwen2.5-coder:7b]
  │ - 프롬프트 (tree-sitter + SymbolSolver + 산출문서 + 본문)
  │ - sequential 호출 (mutex)
  │ - 결과 캐싱 (llm_method_analysis 테이블)
  ▼
[화면: AI 결과 패널 + 적용/취소]

## 마치며

이 시스템의 본질은 **"산출문서가 코드와 일치하는가"** 라는
하나의 질문에 답하기 위한 도구입니다.

- 빠른 정적 분석 (tree-sitter) 으로 **양** 을 처리하고,
- 정확한 의미 분석 (JavaParser+SymbolSolver) 으로 **관계** 를 파악하고,
- 선택적 AI 분석 (LLM) 으로 **의도** 를 보완합니다.

세 도구의 강점만 모아 한 화면에서 **사람이 한눈에 판단할 수 있는 형태로 제공** 하는 게 목표입니다.

결정론적 수치는 신뢰하고, 추론이 필요한 부분은 사용자가 선택적으로 AI 를 활용하세요.
